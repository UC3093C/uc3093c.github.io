Modeling for Software Engineering

What is a Model?

A model is a purposeful simplification of a real-world system that helps us:

  • Analyze complex problems
  • Communicate ideas effectively
  • Test concepts before implementation
  • Document system understanding

Value Proposition

Models bridge the gap between abstract concepts and concrete implementation, making complex software systems more understandable to both technical and non-technical audiences.

How Models Help Us

Today’s Reality

  • Software systems are increasingly complex
  • Human comprehension has limits

Question?

How can you make sense of systems with 40M+ lines of code (like Windows)?

Forward Engineering

Forward engineering: Creating models to design software before implementation.

  • Visualize system architecture and behavior
  • Communicate designs clearly to stakeholders
  • Evaluate plans for clarity, coherence, and quality alignment
  • Guide implementation decisions and priorities

Modeling Mindset

Think first, code second.

Models help us think systematically about software before committing to implementation

Reverse Engineering

Reverse engineering: Creating models to understand existing software after implementation.

  • Document structure and relationships of implemented systems
  • Evaluate architectural quality and identify improvements
  • Communicate system details to key stakeholders: developers, managers, users

Strategic Complexity Management

Models are about managing complexity, through engineering, which ultimately improve outcomes:

  • Models provide simplified representations to manage complexity
  • They bridge the gap between human cognitive limits and system scale

The Modeling Process

Conceptual modeling involves developing abstractions to represent real-world entities, relationships, and rules. It helps structure and communicate knowledge about a domain, making key concepts, their properties, and interconnections understandable to both humans and machines.

What is Modeling?

  • The process of creating structured representations of systems.
  • Focuses on relevant details while omitting irrelevant ones.
  • Helps analyze, understand, and answer specific questions about a system.

Abstraction

  • What It Is: The process of classifying real-world phenomena into concepts.
  • Why It Matters: Bridges the gap between real-world systems and their models.
  • Example: Individual transactions, such as “Payment #XF29853 at 14:32:45,” are grouped under the abstract concept of “Payment”, which represents financial exchanges in a system.

From Reality to Abstraction

  • Phenomenon (Real-World Instances)
    • Concrete, observable events or objects in reality.
    • Examples:
      • “Order #87652 submitted March 10”
      • “Transaction #XF29853 at 14:32:45”
  • Concept (Abstract Model Constructs)
    • Abstract representations that classify related phenomena into meaningful groups
    • Examples:
      • “Order” (a generalized representation of purchase transactions)
      • “Payment” (a generalized representation of financial exchanges)

Concept Structure

In formal conceptual modeling, a concept has three key aspects:

  1. Symbol: The name or representation used to denote the concept
  2. Intension: The defining properties or conditions that determine membership in the concept (meaning)
  3. Extension: The set of all real or possible instances to which the concept applies

Example: “User” Concept

  1. Symbol: User
  2. Intension: An entity with authentication credentials that can interact with the system
  3. Extension: The set of all users who exist or could exist in the system domain

Systems, Models, Views

System

  • The complete solution being developed
  • Encompasses all code, data, and interactions
  • Example: Aircraft

Model

  • An abstraction that represents a system or part of a system.
  • Simplifies complexity while preserving essential characteristics.
  • Can represent different aspects, such as structure or behavior.
  • Example: Scale models, flight simulators.

View

  • Depicts selected aspects of a model
  • Focuses on specific concerns for particular stakeholders
  • Multiple views can represent the same model for different purposes
  • Example: Blueprints, Instrument panel, Electrical schematics

Notation

Notations: A set of graphical or textual rules for depicting views.

  • Provides standardized symbols and conventions (like UML)
  • Enables consistent communication about the system

System-Model Relationship

  • Models are abstractions of systems
  • Multiple models can represent different aspects of the same system
  • Example: Behavioral model and structural model of the same system

Model-View Relationship

  • Views present specific aspects of a model for different audiences
  • Views and models of a single system may overlap
  • Example: Student view shows registration status; Admin view shows all registrations

Creating Effective Models

  • Balance between abstraction and detail
  • Models should answer the specific questions stakeholders have
  • Different notations support different types of analysis and understanding

Types of Models

Computational Models

  • Purpose: Simulate dynamic, time-varying behavior
  • Examples: Weather forecasting systems, physics engines in video games
  • Characteristics:
    • High complexity and development cost
    • Often function as complete systems themselves
    • Require specialized expertise to develop

Analytical Models

  • Purpose: Express quantitative relationships between variables
  • Examples: Financial formulas, performance prediction equations
  • Characteristics:
    • Mathematical in nature
    • Provide precise calculations and predictions
    • Enable “what-if” scenario testing

Descriptive Models (Our Focus)

  • Purpose: Visualize system structure and behavior
  • Examples: Software architecture diagrams, data models, workflow charts
  • Characteristics:
    • Primarily qualitative rather than quantitative
    • Accessible to create and understand
    • Comparable to architectural blueprints

Common Descriptive Models in Software Design

Data Models

  • Focus on information structure
  • Represent entities, attributes, and relationships
  • Examples: Entity-Relationship Diagrams (ERDs)

Application Models

  • Focus on software behavior and structure
  • Represent processes, interactions, and components
  • Primary Example: UML (Unified Modeling Language)

Related UML Modeling Languages

SysML (Systems Modeling Language)

  • Extension of UML specifically designed to supports model-based systems engineering (MBSE) methodologies
    • Broader scope beyond software (hardware, processes, facilities, …)
  • Shares several diagram types with UML but adds system-specific diagrams
  • Enables modeling of complex systems like aircraft, industrial plants, or medical devices

  • Key additions:
    • Requirement diagrams: Capture and trace system requirements
    • Parametric diagrams: Model engineering constraints and physical laws
    • Block definition diagrams: Define system hierarchies and classifications

BPMN (Business Process Model and Notation)

  • Specialized for business processes and workflows
  • Targets business stakeholders rather than engineers
  • Focuses on operational procedures and decision points

Unified Modeling Language (UML)

UML is a standardized family of graphical notations for visualizing, specifying, constructing, and documenting software systems.

Key Characteristics

  • Purpose: Visually represent software architecture and behavior
  • Scope: Primarily for object-oriented systems, but applicable to various software types
  • Structure: Comprises 14 different diagram types, each with specific purposes
  • Flexibility: Adapts to different levels of detail and development phases

Evolution & Governance

  • Origin: First published in 1994
  • Standardization:
    • Became OMG standard in 1997
    • Recognized as ISO standard in 2005
  • Current Standard: Version 2.5 (released 2015)
  • Governing Body: Object Management Group (OMG)

The de facto standard language for software modeling in modern software engineering practice

OMG (Object Management Group) - An international, non-profit technology standards consortium established in 1989. OMG creates and maintains standards for software modeling, middleware, and enterprise integration. When UML became an “OMG standard” in 1997, it meant the specification was formally adopted, would be consistently maintained, and provided a common framework for implementation across the industry.

ISO (International Organization for Standardization) - A global federation of national standards bodies from over 160 countries. ISO standards represent international consensus on best practices and specifications across virtually all industries. UML’s adoption as an ISO standard in 2005 (ISO/IEC 19505) elevated its status from an industry standard to an internationally recognized specification, further solidifying its prominence in software engineering.

UML Diagram Types

Structural Diagrams

Purpose: Visualize the static architecture of a system

Core Structural Diagrams:

  • Class Diagram: Represents the classes, their attributes, methods, and relationships.
  • Component Diagram: Shows the components of a system and their dependencies.
  • Composite Structure Diagram: Illustrates the internal structure of a class and its interactions with other classes.
  • Deployment Diagram: Depicts the physical deployment of artifacts on nodes.
  • Object Diagram: Represents a snapshot of the system at a particular point in time, showing objects and their relationships.
  • Package Diagram: Organizes the system into packages and shows their dependencies.
  • Profile Diagram: Defines custom stereotypes, tagged values, and constraints for UML elements.

Behavioral Diagrams

Purpose: Capture the dynamic interactions within a system

Core Behavioral Diagrams:

  • Use Case Diagram: Illustrates the interactions between actors and the system.
  • Activity Diagram: Represents the flow of activities and actions in a system.
  • State Machine Diagram: Shows the states of an object and the transitions between those states.

Interaction Diagrams (subset of Behavioral):

  • Sequence Diagram: Represents the interactions between objects in a time-ordered sequence.
  • Communication Diagram: Shows the interactions between objects in a network-like structure.
  • Timing Diagram: Represents the timing constraints and interactions between objects over time.
  • Interaction Overview Diagram: Combines activity and sequence diagrams to show the flow of control between interactions.

UML Diagram Selection Guide

For System Architecture:

  • Component, Package, and Deployment diagrams

For Detailed Design:

  • Class, Object, and Composite Structure diagrams

For Process Modeling:

  • Activity, Use Case, and State Machine diagrams

For Object Interactions:

  • Sequence, Communication, and Timing diagrams

Challenges With Using UML

Complexity vs. Practicality

  • UML specification exceeds 700 pages, creating a steep learning curve
  • Grady Booch principle: “For 80% of all software, only 20% of UML is needed”
  • Focus on mastering the core diagrams most relevant to your work

Notation Inconsistency

  • Problem: Varied interpretation and application of UML notation across teams
  • Impact: Reduces diagram readability and increases misunderstandings
  • Mitigation:
    • Establish team-wide notation standards
    • Include legends with non-standard elements
    • Prioritize clarity over notational purity

Time Investment

  • Creating detailed UML diagrams requires significant effort
  • Value proposition: Is the diagram saving more time than it takes to create?
  • Best practice: Match diagram detail level to its communication purpose
    • Quick sketches for temporary discussions
    • Detailed documentation for critical architecture

Maintenance Burden

  • Diagrams become technical debt if not updated with code changes
  • Fast-evolving codebases quickly render static diagrams obsolete
  • Practical solutions:
    • Leverage IDE-based diagram generation tools
    • Focus on stable architectural views
    • Document architectural decisions rather than implementation details

Computer-Aided Software Engineering (CASE) Tools

CASE Tools Overview

Computer-Aided Software Engineering (CASE) tools support various SDLC tasks:

  • Creating UML models and diagrams
  • Reverse engineering code into models
  • Generating code from models
  • Non-UML CASE tools:
    • Data modeling tools (ERD)
    • Business process modeling tools (BPMN)
    • Requirements management tools
    • Analyzing code complexity
    • Calculating software metrics

Available UML Tools

  • 40+ tools in the market according to Wikipedia
  • Proprietary tools:
    • Microsoft Visio
    • IBM Rational Rose
  • Open-source tools:
    • Visual Paradigm
    • PlantUML
    • StarUML

Usage Core Principles

  • Modeling is a creative activity requiring iteration
  • Tools can constrain early creative thinking
  • Start with paper sketches or whiteboard for maximum flexibility

Practical Progression

  • Begin with low-fidelity tools during initial concept exploration
  • Move to digital tools when designs become more stable
  • Consider maintenance requirements for long-term documentation
