How Good is Your Team?

Have you ever heard of SEMA? It’s a fairly esoteric system for measuring how good a software team is. No, wait! Don’t follow that link! It will take you about six years just to understand that stuff. So I’ve come up with my own, highly irresponsible, sloppy test to rate the quality of a software team. The great part about it is that it takes about 3 minutes. With all the time you save, you can go to medical school.

– Joel Spolsky, “The Joel Test: 12 Steps to Better Code”, Agust 9, 2000

The Joel Test is a simple 12-question test that Joel Spolsky, co-founder of Trello and Stack Overflow, created in 2000 to rate the quality of a software team. It’s a quick way to gauge the quality of a software team.

What is The Joel Test?

The Joel Test is a series of quick “yes” or “no” questions. You earn 1 point for each “yes” answer.

Scoring

Score Description
12 A perfect score
11 Tolerable
10 or lower You’ve got serious problems
  • Many software organizations are running with a score of 2 or 3.
  • Big tech companies (e.g., Apple, Microsoft, Meta, …) are running at 12 full-time.

Tip

Joel’s disclaimer is, “you really shouldn’t use it to make sure that your nuclear power plant software is safe.” However, it’s important to clarify that this doesn’t mean it lacks value as a tool for software development teams.

The 12 Questions

  1. Do you use source control?
  2. Can you make a build in one step?
  3. Do you make daily builds?
  4. Do you have a bug database?
  5. Do you fix bugs before writing new code?
  6. Do you have an up-to-date schedule?
  7. Do you have a spec?
  8. Do you have quiet working conditions?
  9. Do you use the best tools money can buy?
  10. Do you have testers as part of the team?
  11. Do you have interview candidates write code?
  12. Do you do hallway usability testing?

Q1: Do you use source control?

Source control is a system that records changes to a file or set of files over time so that you can recall specific versions later. This is a fundamental practice in software development. It’s the first question because it’s the most basic and fundamental tool for software development.

Some benefits of source control:

  • It allows you to revert files back to a previous state.
  • It allows multiple developers to work on a project simultaneously.
  • It allows you to see what changes were made, by who and when.
  • It is an authoritative source of truth for the codebase.

Programmers lack visibility into the work of others, making it difficult to track changes. Reverting mistakes isn’t straightforward. Another advantage of source control systems is that each programmer has a copy of the source code on their hard drive. To date, I’ve not encountered a project utilizing source control that suffered significant code loss.

Q2: Can you make a build in one step?

The question about having a “one-step build” refers to the ability to go from source code to shipping software with a single command or action, emphasizing the importance of a simple, reliable build process. If the build process is complex, it’s likely to be error-prone and time-consuming.

Benefits of a one-step build:

  • It makes it easier to release software.
  • It makes it easier to maintain the software.

Tip

Automation tools and platforms have evolved to support these practices, making it easier to implement a one-step build process. Tools like Make, Ant, Maven, Gradle, Jenkins, and GitHub Actions facilitate automation of the build and deployment processes.

Efficient teams have adopted a single-command process that handles code checkout, compilation, executable generation, and preparation for distribution, whether it’s for cloud or downloads. Processes exceeding one step are error-prone. Near release, the need for quick fixes and final builds is critical. A multi-step process increases the risk of errors and frustration.

Q3: Do you make daily builds?

Frequent builds, ideally daily, are crucial to quickly identify and address any issues, particularly in large teams. Scheduling these builds around a consistent time, like lunchtime, ensures any disruptions are promptly fixed.

Benefits of daily builds:

  • It ensures that the codebase is always in a releasable state.
  • It helps to identify integration issues early, which easies defect diagnosis

Joel notes that when he was a Microsoft on the Excel team their rule was that whoever broke the build, as their “punishment”, was responsible for babysitting the builds until someone else broke it. This was a good incentive not to break the build, and a good way to rotate everyone through the build process so that everyone learned how it worked.

Q4: Do you have a bug database?

Many programmers believe they can mentally track all bugs, which is impractical! It’s easy to forget bugs amid the daily grind. Formal bug tracking is essential.

A bug tracking tool is a platform that not only logs bugs but also manages feature requests, tasks, and more. It acts as a centralized hub for monitoring the status of issues and organizing priorities.

Note

Modern issue tracking systems (e.g. Jira, GitHub Issues, etc.) have evolved to include additional features such integration with version control systems, this allows you to see the source code changes that are associated with a specific issue #

Benefits of a bug database:

  • It ensures that bugs are not forgotten.
  • It helps to prioritize and manage the resolution of bugs.
  • It helps to identify trends and patterns in bugs.

A minimally effective bug database must contain the following information for each bug:

  • Detailed steps to reproduce the bug
  • Expected behavior
  • Observed (buggy) behavior
  • Assignment details (who is responsible for it)
  • Status of the bug (fixed or not)

Q5: Do you fix bugs before writing new code?

This question is about the discipline of fixing bugs before writing new code. It’s a good practice to ensure that the codebase is always in a releasable state. It’s also a good way to avoid the accumulation of technical debt.

The longer the delay in fixing a bug, the more time and money it will cost to resolve. Remember, time is a critical element in software engineering.

Joel recounts how during Microsoft Word for Windows’s development, tight schedules forced shortcuts. One developer simply coded “return 12;” for a line-height function rather than implementing it properly, leaving others to fix it during testing. This practice contributed to the project shipping years late.

Zero Defects Methodology

This does not mean zero defects! It means that at any given time, the highest priority is to eliminate bugs before writing any new code.

Technical debt is a concept in software development that refers to the implied cost of additional rework caused by choosing an easy solution now instead of using a better approach that would take longer – it increases complexity and is more costly in the long term.

Q6: Do you have an up-to-date schedule?

Businesses must make numerous preparatory decisions long before code deployment, such as arranging demos, trade shows, and advertising campaigns. This necessitates maintaining an updated schedule.

Moreover, a schedule is vital as it compels you to define which features will be implemented and encourages you to eliminate the least essential ones, thereby preventing featuritis, also known as scope creep.

Benefits of an up-to-date schedule:

  • It helps to manage expectations.
  • It helps to prioritize work.
  • It helps to manage resources.

Q7: Do you have a spec?

A spec is a document that describes how a system or component should behave. It’s like a blueprint for the software being developed. Writing detailed specs for every feature improves team communication.

Joel’s rule … “no code without spec”.

Programmers are reluctant writers, and they are usually bad at it. They are good at writing code, but they are bad at writing about what the code should do. The best solution is to hire smart program managers who produce the written spec.

Writing specs is like flossing: everybody agrees that it’s a good thing, but nobody does it.

The spec should be a living document that is updated as the project progresses. It should be a detailed description of the feature, including the user interface, the user experience, and the business rules. It should also include a list of test cases that the feature must pass before it is considered complete.

Question: What is a possible solutin that could bridge the gap between writing code and writing what the code should do?

Q8: Do you have quiet working conditions?

Programmers need quiet working conditions to be productive. It is that simple!

Flow or being in the zone is a state of deep concentration where workers are fully immersed in their tasks, leading to high productivity. Achieving flow is crucial for programmers, writers, scientists, etc.

Challenges to Entering the Flow:

  • It typically takes 15 minutes to reach maximum productivity.
  • Fatigue or extensive creative work can hinder entering the zone, reducing productivity for the day.

Distractions and Interruptions:

  • Noise, phone calls, and interruptions, particularly from coworkers, can easily disrupt the flow.
  • A single minute of interruption can result in a loss of 30 minutes of productivity due to the time needed to regain focus.

Q9: Do you use the best tools money can buy?

The best tools money can buy are not necessarily the most expensive. They are the tools that are best suited to the job at hand. Investing in high-quality tools is crucial for enhancing productivity in software development.

Examples of tools that can improve productivity:

  • High-performance computers can save time and prevent distractions during compilation processes.
  • Using two monitors greatly facilitates the development and debugging of GUI applications.

Question: What cost the most in software development?

Imposing unnecessary restrictions, such as limited server space, can significantly waste time and reduce productivity. This is especially critical to consider when the cost of digital storage is quite insignificant compared to the potential loss in productivity that these limitations might bring about. Furthermore, the use of subpar tools not only leads to frustration among programmers but also diminishes their happiness and output. The minor annoyances that arise from inadequate tools can accumulate over time, thereby affecting overall morale and efficiency within a team or organization. Conversely, offering the latest and most advanced tools can serve as a cost-effective strategy to boost morale and retain talent. This approach, focusing on providing high-quality resources, is often more effective than raising salaries when it comes to encouraging loyalty and enhancing productivity among programmers.

Q10: Do you have testers as part of the team?

The role of a tester is to ensure that the software is of high quality. They are responsible for finding bugs and ensuring that the software meets the requirements. The decision to cut corners on hiring testers represents a significant misjudgment of cost savings.

Benefits of having testers as part of the team:

  • They can identify bugs that developers might miss.
  • They can provide a different perspective on the software.
  • They can help to ensure that the software meets the requirements.


Joel’s rule: “Testers are not a luxury, they are a necessity.”, you should have at least one for every two or three programmers.

Tip

Why pay a $100/hour programmer to do work that can be done by $30/hour testers?

Q11: Do you have interview candidates write code?

Would you hire a magician without seeing a magic trick? Or a caterer without tasting their food? Unlikely. Yet, programmers are often hired based on their resumes or how well they can chat about technical trivia, which misses the point.

Traditional Interviewing Flaws:

  • Hiring based on impressive resumes or enjoyable conversations overlooks practical skills.
  • Trivia questions or “AHA!” moments don’t accurately assess a programmer’s ability to code.

Tip

Shift the interview focus from theory and tricks to practical coding tests to better assess candidates’ real-world abilities.

Q12: Do you do hallway usability testing?

Hallway usability testing is a fast, informal way to get feedback on a product. You simply ask people to try it out and watch how they use it. It’s called “hallway” testing because you can do it anywhere you find people – even a hallway!

  • Simple hallway usability tests are free
  • A small group of people (five or six) is sufficient

“you will learn 95% of what there is to learn about usability problems in your code” in a hallway usability test.
